# 📋 v4.0.0 Follow-ups — May 14, 2026

**From:** Claude (Opus 4.7)
**For:** Whoever picks up after the v4.0.0 release lands on `staging` / `main`

These are real gaps surfaced during v4.0.0 regression testing on the developer's machine. They are **not** v4.0.0 blockers — the release ships without them — but each one is concrete enough to drop straight into a task file or implementation plan.

---

## 1. Supabase schema is incomplete

What Tabatha stores in `chrome.storage.local` vs what `tabatha.*` tables exist for:

| Local key | In Supabase? | Notes |
|---|---|---|
| `tabs` (active tab state) | No | Probably should stay local |
| `subGroups` | No | Could sync as a per-user collection |
| `categories` | No | Could sync — user-tunable taxonomy |
| `closedContexts` (archived intents) | No | **Should sync — long-lived data** |
| `sessions` (clock sessions) | Partial via `time_logs` | Org-mode only |
| `sugarBox`, `parkedTabs` | No | Long-lived; probably should sync |
| `urlRules` | No | Long-lived; should sync |
| `tabathaOrg.clients/projects/tasks/operations/initiatives` | **No** | **Highest priority — these are the entities org-mode users will expect on every machine** |
| `pausedIntents`, `inbarNotes` | No | Per-tab ephemeral; local-only is correct |

**Suggested next step:** add a migration `006_add_org_registry.sql` that introduces `tabatha.clients`, `tabatha.projects`, `tabatha.tasks_registry` tables (or one polymorphic `org_entities` table), then extend `syncService` to push these.

## 2. Sync is one-directional (push only)

`syncService.syncToSupabase` writes local → cloud. There is no cloud → local pull. If you create a focus on Browser A and open Browser B, B doesn't see the new focus until you create it locally there too.

Two-way sync is a meaningful design effort:

- Pull cadence: alarm-driven? On-open?
- Conflict resolution: last-write-wins by `updated_at`? Vector clocks? Per-field?
- Migration of existing single-direction state: how do you bootstrap B with A's history without duplicating everything?

Probably a Plan-025-sized piece of work. Likely Tier-1 priority for any multi-machine user.

## 3. Tab groups are tracked but never displayed

`groupService` listens to `chrome.tabGroups.*` events and writes to `chrome.storage.local.subGroups`. No Tabatha UI surface shows the result. A user creating a Chrome group via Chrome's built-in UI has no way to see "Tabatha sees these N groups."

Suggested next step: a "Tab Groups" section in the Tabs panel of [home/index.jsx](src/home/index.jsx) or sidebar — collapsible list of group names with member-tab counts.

---

## 4. Companion bridge has no profile scoping

`companionService` connects to a hardcoded `ws://localhost:9147` with no profile/session ID. Running Tabatha in two browser profiles simultaneously will cause:

- Both clients receive identical companion broadcasts (`TICK`, `IDLE_STATE`, `CLOCK_BREAK`).
- Both clients send clock-in/clock-out events — last write wins, time tracking gets corrupted.
- Idle/desktop activity gets mirrored to both, double-counting.

**Workaround for now:** stop the companion app when regression-testing in a QA profile.

**Real fix:** scope the WS connection by browser-profile ID (already stored in `chrome.storage.local.browserProfileId`). Probably requires companion-app changes too — depends on how the companion is structured.

## 5. Context drift → webhooks

Currently `tabTrackingService.handleTabActivated` detects context drift (you said you'd focus on X but switched to Y) and broadcasts `CONTEXT_DRIFT_DETECTED` internally. The user wants this also fired through the existing webhook system (`fireWebhook` in `src/background/webhooks.js`) so external tools can react.

Small addition — likely one call to `fireWebhook('context_drift', { ... })` at the drift-detection point.

---

## 6. Focus / funnel flow discoverability

User feedback during regression: "Set focus → add items → switch → complete → funnel stage transitions — I think it works because I only think I did it right. Not sure what I was expecting."

This is a UX/onboarding gap, not a bug. The funnel stages (`unsorted → todo → focus → addressing → resolved → roadblocked`) are powerful but invisible to new users. Suggestions:

- First-run tooltip/walkthrough on FocusBar
- Inline explanation in the funnel-stage picker (each stage has a one-line "what this means")
- Or: simpler default mode that hides funnel stages until the user explicitly opts in

---

## 7. Two-browser sign-in clarification

User signed in on two browsers; not clear what data sharing should look like.

What's true right now:

- Supabase data (profile, orgs, teams, synced focus_items, intent_history) is per-user — both browsers see the same.
- `chrome.storage.local` is per-extension-installation — tabs/focuses/sessions don't auto-mirror.
- One-way sync means Browser A pushes; B doesn't pull.

Action: surface this clearly in Settings → Account → Sync Status, perhaps with a per-browser-profile name picker so users can tell their machines apart. Stable browser-profile-id already exists in `chrome.storage.local.browserProfileId`.
