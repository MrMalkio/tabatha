# Cortex — Agent Data Map (C14)

Status: populated (Fable overnight, 2026-07-10)
Parent: [Program Spec](./00-cortex-program-spec.md) §9 · Governance process: [C14](./features/C14-agent-data-map-governance.md)
Last updated: 2026-07-10 (Fable overnight)

This is the **authoritative catalog** of every signal Tabatha/Cortex captures — Cortex-native and
pre-Cortex alike: where it lives, which partition it belongs to, how long it is retained, its
redaction state, and the access contract for downstream agents. **Every agent leveraging Tabatha
data reads this map before touching a signal.** Per C14, this file documents *current state, not
history* — when a signal's storage/retention/redaction changes, the row is updated in place, not
appended-to. Every row here traces back to a real file read during this session (paths cited); no
row is a guess.

**Governance reminder (see C14 for full process):** this map is a gate for merging new capture
surfaces, not documentation-for-its-own-sake. Any PR adding a new `kind`/`surface` to the
Observations Ledger, a new "Cortex" settings key, or a new Supabase migration touching capture data
must add or update a row here in the same change.

Columns (frozen schema):

| Signal | Source | Storage (local/cloud) | Partition (personal/org) | Retention | Redaction | Agent access |
|--------|--------|-----------------------|--------------------------|-----------|-----------|--------------|

## Open questions this pass surfaced (not resolved here — see C14 §"Open questions" for the governance-process versions)

1. **No true personal/org partition on companion or Supabase-sync data.** Only the Cortex ledger (`cortexLedger` → future `cortex_observations`) implements the clock-state-based partition rule from program spec §6. `companionRecentSessions`/`desktopActivity` and all of migration 008's tables are profile-scoped but not clock-time-partitioned — so "personal never flows to org" is enforced for Cortex observations but **not yet** for the broader desktop-activity/clock-history data that predates Cortex. This is a gap between the privacy spine (spec §6) and current reality, not a documentation error.
2. **Orphaned capture frames.** `pruneLedgerByAge()` deletes ledger *records* but never the JPEG files they referenced. Over time this can leave redacted-away or expired frames sitting on disk indefinitely under `Downloads/<root>/<partition>/<month>/`.
3. **OS-level file access is an unenforced boundary.** Capture frames and ledger exports are plain files under the user's Downloads folder — any local process (agent or otherwise) with filesystem access can read them regardless of what this map's "Agent access" column says. The map documents the *intended* contract; only OS file permissions (out of Tabatha's control) enforce it in Phase 1.
4. **Companion WS bridge has no auth token.** `ws://localhost:9147` accepts any local connection. Low risk today (loopback-only) but worth a row update if that ever changes.
5. **`storage.logsCap` is dead configuration.** `logger.js` hardcodes its own 500-entry cap independently of `DEFAULT_SETTINGS.storage.logsCap`; they agree today by coincidence, not by wiring.
6. **`tabatha-mobile` claim correction.** See §5 above — flagged explicitly so whoever next touches Cortex/mobile scoping doesn't inherit C14's "substantially built" characterization uncritically.

These are handed to whoever picks up T4 (real cloud-batch push for the Cortex ledger) and any later mobile-integration task — not blocking for Phase 1 sign-off, but real enough that C14's "living contract" principle says they belong in the map now rather than being rediscovered later.
