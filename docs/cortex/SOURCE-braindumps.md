## Dump 1 — Capture surface

> Options 1, 2 and 3. The browser extension should capture screenshots of the tabs after a user goes into a new tab, and if a user doesn't navigate away from a window but is just working in a window then maybe it takes new screenshots every five or 10 seconds — the time frequency can be adjusted in configuration. When screenshots are being taken inside the browser then the desktop companion won't need to take additional screenshots all the time, but when the user is effectively moving over to another application that's outside the browser the desktop companion should take over the screenshots. This also needs to take into consideration when a user has more than one screen — it's not always going to be a matter of just capturing screenshots every five seconds; sometimes it's going to be a means of just taking a screenshot because a user changed context or moved from one window to the next window. Because we've already built a system that captures a bunch of other context, we don't always need a screenshot, and instead we should use the context that we have to decide when a new capture should take place. As for storage, we should make it so images can automatically archive back into external storage such as Google Drive, OneDrive, or an external hard drive if available. Even if those aren't available it can be configured where these files will be stored on the system. Along with this and everything else that Tabatha captures, there needs to be an updated map for the agents that will leverage this information and all the other information that existed before capturing became a thing. Also capturing needs to be able to be turned off for different users if they want to turn it off; and in the case of managing captures for team members when they are clocked in, organization admins need to be able to require or even mandate that screen capturing is turned on once they clock in. For those using this application independently they can fully control that for their personal profile, and when they are still capturing for personal means it does not go to the organization — instead those are saved and archived to a different place, and they both have retention plans separate from organization-level work (which is everything when someone is clocked in). That retention plan is decided by the organization admin and can be based around both time and available space on a machine. We might as well also include manual screen recording — if a user wants to record a video with sound from the machine and from their mic — both from the extension and from the desktop, leveraging what already exists inside Chrome and inside the computer. Currently our desktop companion is only on PC but our extension works for both PC and Mac. Either way we need to account for both PC and Mac ultimately, but for right now we are prioritizing PC. We also want multiple configuration options on when all the data is processed — whether it's done multiple times throughout the day at low levels and then higher levels at the end of the day so it helps guide what's going on the next day, or can give intel to another autonomous agent that a user might have to just let them go ahead and make changes so that the next day the user comes back to new things the agent put together — completing knowledge work, building a new extension, creating a dashboard, whatever it may be.

## Dump 2 — AI routing / autonomy

> Let's have all of them honestly. But another method not mentioned is creating cron jobs inside a user's active harnesses they may already have — more manual on the user side, more likely for a personal user, since all the data is stored on the user's computer first and then later moved to a different storage location. If the user has Claude Code or Codex or something of the like installed, then Tabatha should be able to set up a script or scheduled task from within Tabatha that lives in those harnesses. This can be done directly from within Tabatha or the desktop companion reading the folders of those different harnesses. And integrate with what we're already working on with integrating Tabatha into Headbox, which already does a good job of understanding what's going on inside the different harnesses somebody has installed and giving some governance. Take all these methods, start with the easiest one first (the cron-in-harness one), and have the others as escalating levels — think in terms of both personal and professional, individual and teams. Our job might also be creating the master system prompts for the task of optimization, surfacing results in a dashboard, clicking yes/no on what should be created, then having Tabatha create the new prompt using whatever AI it has access to and triggering it as a task (reactive vs proactive). Consider how proactive the agents are with this info — could be left to the user but also configurable in the platform. Unrelated: take into consideration the same level of tracking for a mobile phone. Tabatha needs the ability to let a user replace virtually every button/input with an audio button to tell Tabatha what's going on, then Tabatha decides what to do — because we have a lot of buttons and inputs that should remain available for manual management or certain interaction levels that help working style; but many users need Tabby to be as passive as possible, almost invisible, constantly updating its own records of what's happening, what tabs are linked to what, what intents really are, how long something has actually been worked on — assuming the human is always lacking on updating and management. Looking at phone records/call logs, when emails were replied to, when text messages were replied to, when the computer was off and on, knowing when an AI agent is the one controlling a browser/window/computer so attention and activity is attributed to the right entity — so Tabatha provides better info on how well the user is optimizing and leveraging their resources and tools.

## Dump 3 — Privacy posture

> Local-first opt-in with cloud batch as backup — a mix of the two. Fully local with local-in-companion is not V1; we don't have the capacity to stuff a local model inside our companion yet, and we haven't even launched this first version to anyone other than myself. But ultimately we want all the options, and the user can configure as needed for personal, and the organization can configure as they want for non-personal. To begin, everything is stored on machine. There should be configuration especially for sensitive work styles — e.g. finance or legal can decide if screenshots can happen or be automatically deactivated for certain types of work, set it up so certain websites deactivate screenshots at that moment. E.g. if QuickBooks is open for a client and the user doesn't want QuickBooks captured, then when QuickBooks is the focused tab/window it would not take the screenshot of that tab — but it would not deactivate screenshots of other tabs when QuickBooks is not in focus. And I wonder if we can set up auto-redaction at capture time, such as blurring the bottom 80% of a QuickBooks tab before the screenshot, so the text of which page / which client is preserved — because it's the combination of client information and client private information (their identification and their private information at the same time) that creates the privacy issue, not "what client was I working on." Output all brainstorming and ideas into a Google Doc so it's not locked in the thread. Also the ability to potentially capture specific windows — capturing the entire screen, capturing different screens as different images at the same timestamp, and capturing a specific window by itself (especially with an ultrawide monitor + another monitor of a different aspect ratio, where screenshots can come out weird and the agent may not know the layout). Look at audio input the same way — a WhisperFlow clone/integration, or an open-source version we build on top of as a plugin, where everything that goes into the dictation app also goes into the Tabatha context. Or build our own from scratch since other apps just wrap different models; we can still use their code as inspiration.

## Dump 4 — Voice output + hotkeys + operational asks

> One thing I forgot: audio output as well — allowing Tabatha to have a voice that can speak to the user in place of some of the pop-ups/modals they have to engage with to inform and direct Tabby. The AI layer would have two-way communication, and in some cases a moment where a modal would have happened is actually a moment where Tabby just wakes up and speaks — not pre-recorded, the same every time — instead something short, very soft, not super interrupting. Maybe an audible tone lets the user know Tabby is about to say something, and before Tabby speaks the mic picks up what the user might be saying, giving the user just enough time to tell Tabby to hold off, and Tabby responds "ok I'll come back later." Otherwise (or if that sub-feature is deactivated) Tabby just speaks contextually based on the type of modal that would have happened. If the user doesn't respond or isn't at the machine, that's when a modal pops up instead. The user always has a hotkey for Tabby to just speak. Two main hotkeys — one to just speak to Tabby, the other for transcription — so audio can use different models (transcription can use a local/cheaper model; thinking uses something more expensive). Maybe both use the same transcription model but one only transcribes and the other transcribes and sends for thinking. A third hotkey is for making notes — like transcription but not for a specific input: the first hotkey is field transcription, the second is speaking to Tabby (awake, likely real-time conversation, with levels: real-time convo; speak → process → output back; or speak → Tabby updates something inside Tabby/Flux/context management), and the third is literally a voice note stored inside the context (more related to the Flux system, of which Tabby is a subset).
