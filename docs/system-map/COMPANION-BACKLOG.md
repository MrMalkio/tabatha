## (A) Blocking team rollout

1. **`feat/companion-update-manifest` (updater manifest, companion v0.2.1) is unmerged, stacked on Koda's widget branch** — blocks the companion's Supabase-Storage auto-update from actually delivering the hardened build to team machines; needs a carve-out/cherry-pick. — `docs/system-map/SYSTEM-MAP.md` §3.1/§5.4/§7.3, `docs/progress.md` 2026-07-18 entries, Asana [1216679069945012](https://app.asana.com/1/9526911872029/project/1214031898449333/task/1216679069945012)
2. **`dist/` "reload landmine"** — Chrome's pinned load-unpacked path may be serving unreviewed 6.8.2 widget code instead of the believed 6.7.22 companion-adjacent line; the companion's atomic-replace updater manages `%APPDATA%\Tabatha Desktop\extension\`, not the raw dev `dist/`, so what the companion ships and what Chrome actually runs can silently diverge. — `docs/system-map/SYSTEM-MAP.md` §3.1, `.headbox/sticky-notes/2026-07-01-extension-companion-vanish-incident.md`
3. **Extension update-channel swap-step bug** (`fix/updater-swap`, 6.7.24, unmerged) — prod machines sit one version behind the advertised channel, undermining the whole "companion drives updates" model. — `docs/system-map/SYSTEM-MAP.md` row 1, §7.3
4. **Extension `.pem`/persistence bug** — the unpacked extension can disappear on machine restart, forcing a manual "Load unpacked" redo; daily friction for the only active dogfooder and a real data-loss risk, which directly undercuts the companion's "dummy-proof guided install" promise. — Asana [1216437510221814](https://app.asana.com/1/9526911872029/project/1214031898449333/task/1216437510221814)
5. **Companion autostart + crash-hardening fixes exist in code but haven't been redistributed** (NB-10, affects Malkio's own OD machine) — autostart-default-ON (commit `783bed2`) only takes effect on a fresh installer run, and the Mutex-poison crash fix (`efb6e59`) predates the last MSI build; no rebuilt installer has shipped since either landed. — Asana [1216279590361708](https://app.asana.com/1/9526911872029/task/1216279590361708)
6. **Code signing is inactive** — the installer is unsigned and triggers Windows SmartScreen "Unknown publisher," a trust-eroding first impression for Reggie/Po's install call. Pipeline is fully wired on `feat/code-signing` (`docs/CODE-SIGNING.md` in `tabatha-desktop`); blocked purely on certificate purchase (human-gated, not engineering). — `tabatha-desktop` repo scan
7. **Chrome Web Store vs. companion silent-update conflict is undecided** — once Chrome's own auto-update takes over (CWS listing), the companion's silent-update mechanism needs a resolved relationship (coexist / replace); currently an open question blocking a clean distribution decision. — `docs/CHROME-WEB-STORE-LISTING.md`
8. **Companion↔extension WebSocket auth is Stage 1 only** (origin-gating, commit `76a4017`) — the extension's own v6.7.22 changelog discloses this as a not-fully-fixed security issue (any web page could otherwise connect and trigger OS capture); rolling a broader build out to team machines before Stage 2 handshake auth lands increases exposure. — `tabatha-desktop` repo scan (`ws_server.rs`), extension `changelog.json` v6.7.22 entry
9. **No confirmed GitHub remote for `tabatha-desktop`** — flagged as a suspected backup/continuity risk pending Argus's in-progress companion-state survey. — Asana [1216679069896234](https://app.asana.com/1/9526911872029/project/1214031898449333/task/1216679069896234)
10. **Companion bridge regression coverage is stale** — parking-lot pre-production checklist explicitly called out "companion bridge" as untested in the full manual regression matrix since Plans 027/028 landed; no later record found confirming that pass ever happened. — `.headbox/parking_lot.md` 2026-05-26

## (B) Reliability / bugs

1. **Companion bridge has no browser-profile scoping** — hardcoded `ws://localhost:9147` with no session/profile ID means two Chrome profiles both receive identical companion broadcasts (`TICK`/`IDLE_STATE`/`CLOCK_BREAK`), causing last-write-wins clock corruption and double-counted idle/desktop activity. Flagged independently in three sources since v4.0.0 and still open. — `.headbox/sticky-notes/v4.0.0-followups.md`, `Tabatha_Changelog.md` v4.0.0 known-issue note, `docs/FEATURE-MATRIX.md` "Cloud sync push" row
2. **WS Stage-2 handshake auth still pending** (see A8) — needs a coordinated companion + extension-bridge fix per the (not-in-this-worktree) security doc `docs/security/companion-socket-auth.md`.
3. **NB-10 unresolved in the field** — the autostart/crash fixes are written but not verified working on the one machine that reported the symptom; "fixed in code" ≠ "fixed for the user" here.
4. **B08: autopause still fires on non-browser activity** — status "🔧 Fixing"; needs the companion `APP_SWITCH`/`isRecentlyActive` signal wired to a confirmation-prompt UX ("Still on X?") instead of a hard autopause, per the user's explicit request. — `docs/features/B08-autopause-false-trigger.md`
5. **FIX-01…FIX-07 shipped in code (v6.5.0) but every matching Asana task is still open** — Integrations-panel false-negative, tray clock in/out no-op, Activity-Log broken-terminal crash, silent "Check for Updates," Desk-panel clock not reflecting in the extension, random Mutex-poison crashes, UI-copy rename. Needs a verify-and-close pass; FIX-06 specifically needs field re-verification given B/3 above. — `Tabatha_Changelog.md` v6.5.0; Asana [1216220528621209](https://app.asana.com/1/9526911872029/task/1216220528621209) / [1216220498940315](https://app.asana.com/1/9526911872029/task/1216220498940315) / [1216220514372964](https://app.asana.com/1/9526911872029/task/1216220514372964) / [1216220514466770](https://app.asana.com/1/9526911872029/task/1216220514466770) / [1216220499283880](https://app.asana.com/1/9526911872029/task/1216220499283880) / [1216220514453897](https://app.asana.com/1/9526911872029/task/1216220514453897) / [1216220439608899](https://app.asana.com/1/9526911872029/task/1216220439608899)
6. **Idle-auto-break doesn't honor "Yes, on task"** — a forgotten open Zoom/Meet tab used to disable idle detection all day (fixed v6.3.1), but the auto-break still fires after 5 min Chrome-idle even after the user confirms via the companion-fed `IDLE_PROMPT` — should suppress in that case. — `.headbox/sticky-notes/plan-036-037-038-handoff.md`
7. **Desktop-activity sync payload shape is unvalidated** — stored as a defensive raw payload rather than a validated schema, a latent data-quality risk for anything downstream (Cortex capture, reporting). — `.headbox/sticky-notes/supabase-sync-batch1-handoff.md`
8. **`uptime_ms` always reads 0** — open since the original 2026-05-10 status note, no later mention found confirming a fix. — `.headbox/sticky-notes/desktop-companion-status.md`
9. **macOS/Linux window-monitoring and idle-detection are unimplemented stubs** — companion is Windows-only in practice despite `#[cfg(target_os)]` scaffolding for all three platforms. Not urgent for a Windows-only team, but a hidden platform gap. — `tabatha-desktop` repo scan (`window_monitor.rs`)
10. **Companion is out of sync with newer extension features** (backburner state, focus priority, calendar sync) — flagged 2026-05-28, no dedicated companion-feature-sync plan created since. — `.headbox/parking_lot.md` 2026-05-28
11. **Cross-profile "divergence" attribution deferred by design** — if a user is active in a Chrome profile without the extension, the companion can't cleanly attribute that activity; documented as a known limitation, not solved. — `docs/superpowers/plans/2026-06-29-tabatha-first-deployment-plan.md`
